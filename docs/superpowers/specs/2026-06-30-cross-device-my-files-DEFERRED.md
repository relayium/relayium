# 跨设备"我的文件"（Stored Transfer Mode 2）— 设计备忘（暂缓）

**状态：DEFERRED（已确定方向，暂不实现，以后再做）**
日期：2026-06-30

## 目标

用户用"跨网络存储"上传文件后，在**另一台设备登录账号**即可看到并直接下载自己上传的文件——
无需原始下载链接里的 `#k=` 密钥片段。

## 核心矛盾（不可绕过）

零知识加密 与 跨设备自动解密本质互斥：

> 只要"凭一个登录 cookie，服务器就能把解密密钥交给新设备"，
> 就等于服务器自己也持有密钥——零知识当场失效。

凡是服务器能"验证"的凭据（登录密码、TOTP 2FA 种子、魔法链接、OAuth），
服务器都"知道"它，因此都**不能**充当零知识密钥。

## 已选方向：A1 —— 独立"保险库口令" + 一次性恢复码

- 用户自设一个**与登录方式无关**的密码短语（口令），仅在浏览器本地用于派生密钥，
  **永不发送给服务器**。
- 派生：`口令 ──Argon2id──> 32B 保险库主密钥`；主密钥在浏览器里"包住"（wrap）每个文件的随机密钥；
  服务器只存「被包住的密文密钥 + 文件密文」，两者都看不懂。
- 新设备登录后**再输一次同样口令** → 本地重算主密钥 → 解开文件密钥 → 解密。
  浏览器记住主密钥，直到登出/清缓存。
- 设置口令时系统生成**随机恢复码**（如 `RELAYIUM-7K2F-...` 或 12 词助记词），
  抄下来兜底遗忘。**口令与恢复码都丢 = 文件永久不可解**（零知识的代价 = 卖点）。

### 体验
- 设备 A 首次用"我的文件"：设置口令 → 显示恢复码 → 此后上传时本地加密。
- 设备 B 异地登录：看到列表（文件名也是加密的，本地解出显示）→ 首次输口令解锁 → 下载即本地解密落盘。
- 「登录证明你是你；保险库口令证明只有你能读」——两层分离，偷了登录态也解不开。

## 未选但记录在案的替代方案
- **A2. Passkey / WebAuthn PRF**：用同步 passkey 一点即解、不用记口令；真零知识；
  但依赖跨设备同步的 passkey 生态（iCloud 钥匙串 / Google），实现面更大。可作为后续增强。
- **B. 服务器保管密钥**：登录即下、零额外步骤；但服务器能解密这部分存储文件，
  放弃这块的零知识。与项目隐私立场冲突，未选。

## 待实现时需要细化的点
- 口令强度门槛与校验 UI。
- 主密钥在内存中的留存策略 / 登出时清除。
- 文件名/元数据的加密展示（manifest 已加密，需在列表页本地解出）。
- 恢复码生成、展示、"已保存"确认、用恢复码重设口令的流程。
- 与现有 Mode 1（一次性下载链接）共存：同一份密文能否两种方式都取？还是分库？
- 服务器侧：被包住的密钥存储、按账号列出文件的接口（owner-gated）。
