# Relayium 网页版 M0（MVP）设计文档

- 日期：2026-06-28
- 状态：已评审通过，待写实现计划
- 代号：Project Relay / Relayium

## 1. 背景与定位

Relayium 的目标是做一个**严肃开源**的「下一代智能文件传输协议」：用户只按一个
「发送」，软件自动选择最佳传输路径（局域网直连 → P2P → Relay → 临时云存储），
**默认端到端加密**，密钥只存在于发送端与接收端。

本项目由更早的明文临时文件分享服务（AuraShare）演进而来——一次产品风险分析指出
AuraShare 最大的问题是没有端到端加密、服务器能看到明文，因此转向「E2E + 自动选路」
的协议化方向。

**本文件只覆盖 M0（第一阶段 MVP），形态为网页版。** M1–M3 见第 8 节路线图。

### 1.1 形态决策：网页版优先

用户选择从网页版做起（而非 CLI）。需要明确接受一个硬约束：

> **网页版做不到「零服务器」。** 浏览器不能做 mDNS / UDP 广播（无法自行发现局域网
> 设备），且浏览器之间直连只能用 WebRTC，而 WebRTC 建连必须有信令服务器交换
> SDP/ICE。因此网页版 M0 必然包含一个轻量信令服务器。

代价是放弃「零服务器」，换来「打开网页即用、无需安装」。**文件内容仍是端到端加密、
永不经过服务器**，E2E 原则不受影响。该信令基础设施也为 M2（跨网 Relay）提前铺路。

### 1.2 竞争与差异化

网页版的直接对手是 **Snapdrop / PairDrop**（开源、流行，已实现「浏览器 + WebRTC +
局域网风格 + E2E」）。Relayium 网页版的差异化：

1. **应用层端到端加密 + SAS 校验**：不只依赖 WebRTC 的 DTLS（其指纹经信令服务器
   交换，恶意服务器可中间人调包）；在 DataChannel 之上再做一层 X25519 + AEAD，密钥
   永不上服务器，并用短验证码（SAS）防中间人。这是比 Snapdrop 更彻底的 E2E。
2. **持久设备身份**（M1 落地）：固定的几台设备一次配对、永久互信，无需每次交换码。
3. **同一套协议未来接 CLI / 移动端**（M3）：Snapdrop 永远只是网页。

## 2. 范围（Scope）

### 2.1 M0 必做（In scope）

- 轻量信令服务器（Go + WebSocket），按公网 IP 把设备分到同一「房间」。
- 网页客户端（Svelte 静态 SPA）：显示同房间设备、拖拽选文件、选目标设备、发送。
- WebRTC DataChannel 直传文件（文件不经过服务器）。
- 应用层 E2E：X25519 ECDH 协商会话密钥 + AES-256-GCM 分片加密。
- SAS 短码校验（跨网默认开启，同房间 LAN 可默认跳过）。
- 大文件流式落盘（不把整文件读进内存）。
- 传输进度显示 + 接收端 SHA-256 完整性校验。

### 2.2 M0 不做（Out of scope，留给后续阶段）

- TURN 中继 / 跨复杂 NAT（M2）。
- 账号体系、设备持久配对、离线暂存（M1/M2）。
- CLI、移动端原生（M3）。
- 断点续传、目录、多文件批量、自动解压（M1）。
- `send stdin` / docker / 剪贴板等多对象类型（M3）。

## 3. 架构

```
┌─────────────┐  WebSocket(信令)  ┌──────────────────┐  WebSocket(信令)  ┌─────────────┐
│  浏览器 A   │◀────────────────▶│   信令服务器(Go)  │◀────────────────▶│  浏览器 B   │
│ (发送端)    │                   │  按公网IP分房间    │                   │ (接收端)    │
└──────┬──────┘                   │  只转发信令/元数据 │                   └──────┬──────┘
       │                          └──────────────────┘                          │
       └────────────── WebRTC DataChannel（文件直传，端到端加密）─────────────────┘
                                文件永不经过服务器
```

### 3.1 部件职责

| 部件 | 职责 | 不做什么 |
|---|---|---|
| 信令服务器 (Go) | WebSocket 接入；按公网 IP 分房间；广播房间内设备列表；转发 offer/answer/ICE | 不碰文件内容；不存任何明文；不做账号 |
| 网页客户端 (Svelte) | 连信令、显示设备、发起/接收 WebRTC、应用层加解密、流式落盘、进度与校验 | 不做设备发现的网络层（交给服务器房间机制） |
| STUN | M0 用公共 STUN 获取候选地址 | M0 不含 TURN |

### 3.2 分层（关键原则：加密层与传输层解耦）

```
discovery(房间/设备列表)
  → identity(本会话临时密钥对 X25519)
  → session(ECDH 协商会话密钥 + SAS 校验)
  → transport(WebRTC DataChannel；未来可插 LAN-TCP/Relay)
  → transfer(分片、AEAD 加密、进度、SHA-256 校验、流式落盘)
```
**先协商密钥、再选路**；每条传输路径复用同一加密层，不重复造加密。

## 4. 数据流（同公网 IP / 局域网场景）

1. A、B 打开网页 → WebSocket 连接信令服务器 → 服务器按公网 IP 分到同一房间 → 双方
   收到房间设备列表，互相可见。
2. A 拖入文件、选择 B → A 创建 WebRTC offer（含本端 X25519 公钥）→ 服务器转发给 B。
3. B 回 answer（含本端 X25519 公钥）→ 交换 ICE 候选 → DataChannel 打开。
4. 双方各自用对端公钥 + 本端私钥做 ECDH，得到共享会话密钥。
5. （可选）双方屏幕显示由会话密钥派生的 SAS 短码；用户确认一致 → 防中间人。
6. A 把文件按固定大小分片，每片 AES-256-GCM 加密后经 DataChannel 发送；B 解密、
   边收边算 SHA-256、流式写盘。
7. 传输完成，B 校验 SHA-256 与发送端声明的哈希一致 → 提示成功。

## 5. 安全设计

- **威胁模型**：假设信令服务器可能被动观察、甚至主动中间人；网络可被窃听。要求：
  服务器无法读取文件内容，也无法在用户确认 SAS 的前提下中间人。
- **密钥协商**：每次传输生成临时 X25519 密钥对（M0 不持久化身份；持久身份属 M1）。
  ECDH 得共享密钥，经 HKDF 派生会话密钥。
- **加密**：文件分片 AES-256-GCM（每片独立 nonce，含分片序号防重排/重放）。
- **完整性**：每片 GCM tag + 整文件 SHA-256 端到端校验。
- **防中间人**：SAS 短码（由会话密钥派生，如 6 位数字或 emoji 序列）。跨网默认开启；
  同房间 LAN 可默认跳过（信任本地网络），但提供开关。
- **元数据最小化**：服务器仅见房间归属、设备昵称、在线状态、信令消息；文件名在 M0
  随首片元数据经加密信道传输，不暴露给服务器。

## 6. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 信令服务器 | Go + `coder/websocket` | 单二进制、轻量；契合用户 Go 背景 |
| 客户端框架 | Svelte（静态 SPA） | 产物小、无虚拟 DOM 开销、适合「打开即用」 |
| 客户端加密 | libsodium.js (WASM) | X25519 + AEAD 一站式，比裸 WebCrypto 省心且不易出错 |
| 传输 | WebRTC DataChannel | 浏览器间唯一可行的 P2P 通道 |
| 大文件落盘 | File System Access API（Chromium）+ StreamSaver 兜底 | 避免整文件进内存 |

## 7. 验收标准（量化）

M0 视为完成需全部满足：

1. **互通**：同一局域网（同公网 IP）下两个浏览器能互相发现并成功传输文件。
2. **大文件不爆内存**：1GB 文件传输全程内存占用受控（流式落盘，不整体缓冲）。
3. **完整性**：接收端文件 SHA-256 与发送端一致。
4. **真 E2E**：抓包确认文件数据只走 DataChannel；信令服务器只见信令/元数据，无文件内容。
5. **防中间人**：模拟恶意信令服务器调包密钥时，两端 SAS 不一致、可被用户发现。
6. **浏览器矩阵**：Chrome 全功能通过；Firefox/Safari 至少基本传输通过（大文件落盘可降级）。

## 8. 后续路线图（非本 spec 范围）

- **M1 开发者体验 + 持久身份**：目录/多文件、断点续传、并发分片、持久设备配对
  （「指名直发，无需每次配对」）。
- **M2 跨网 Relay**：TURN/中继、NAT 打洞、目标离线时临时密文暂存。**此阶段起有服务器
  带宽成本，需在路线图中诚实标注成本模型。**
- **M3 协议标准化 + 多端**：把线协议写成规范，接 CLI / 移动端复用；`send` 扩展到
  stdin / docker 镜像 / 剪贴板等多对象类型，迈向「开发者之间的 TCP」。

## 9. 主要风险

1. **大文件内存与浏览器兼容**：File System Access API 仅 Chromium 完整支持；
   Safari/Firefox 需 StreamSaver 兜底，体验有差。M0 明确「主推 Chrome、其它兜底」。
2. **同网客户端隔离**：部分 AP 开启 client isolation，同房间也连不通 → 需回退提示。
3. **iOS Safari WebRTC 兼容坑**：移动端浏览器单独验收。
4. **WebRTC 复杂性**：连接建立、ICE 失败、DataChannel 背压（大文件需做流控）是主要
   工程难点，需在实现计划中单列。
